## How to use this index

Each entry lists:
- **Purpose** — one-line description of what the system does
- **Source of Truth** — the authoritative DB table, file, or module
- **Details** — where the full specification lives
- **Do not recreate** — the most common mistake to avoid

Status markers: ✅ implemented · ⚠️ needs documentation · 🔜 planned (no backend yet)

## A — Authentication & Identity

### 1. Auth / Login / Registration
**Purpose:** Register and authenticate users (emp / co / edu); bcrypt passwords; post-login routing by user_type.
**Source of Truth:** `users` table · `auth.py` (`hash_password`, `verify_password`) · `index.auth.js` (`redirect()`)
**Details:** `CLAUDE.md → Authentication System` · `CLAUDE.md → Auth Gateway Rules` · `ARCHITECTURE.md §46`
**Do not recreate:** Do not add a second login form or second redirect logic. `redirect(u)` in `index.auth.js` is the single post-login routing authority.

---

### 2. JWT Token System
**Purpose:** Issue and validate signed JWT tokens for all API calls; payload carries `user_id`, `user_type`, `country_code`.
**Source of Truth:** JWT generation in `auth.py`; algorithm HS256; token stored in `localStorage.tw_jwt`
**Details:** `ARCHITECTURE.md §46`
**Do not recreate:** Do not pass `user_id` in request body — always extract from `token.get("user_id")` server-side. Do not use `X-User-Id` header — Bearer token only.

---

### 3. Public ID / tw_id System
**Purpose:** Unique identifier for every user: `{PREFIX}{COUNTRY_CODE}{10_HEX}`. U=emp, C=co, T=edu.
**Source of Truth:** `users.tw_id` (UNIQUE) · generated by `generate_tw_id(user_type, country_code)` in `auth.py`
**Details:** `CLAUDE.md → User ID Format` · `CLAUDE.md → Smart Public Profile Router Rules §9`
**Do not recreate:** Do not create per-entity ID generators. Future entity public IDs (J/P/A/V/D/E/L/Q/S) use `generate_public_id(prefix)` — no country code in the ID.

---

### 4. Smart Public Profile Router · `/u/{tw_id}`
**Purpose:** Unified public URL for all account types. Routes to employee / company / education profile based on `users.user_type` in DB.
**Source of Truth:** `users.user_type` · `get_user_info_by_tw_id(tw_id)` in `auth.py` · Smart Router route in `server.py`
**Details:** `ARCHITECTURE.md §63` · `CLAUDE.md → Smart Public Profile Router Rules`
**Do not recreate:** Do not create separate public routes like `/company-public/` or `/edu-public/`. Do not route by tw_id prefix alone — always query the DB.
**Legacy redirect (PR #386):** `/company-profile` is a legacy redirect only — it checks `user_type==="co"` → `/u/{tw_id}`; non-co users → `/home`; no JWT → `/login`. `/company-profile?id=123` → server-side 302 → `/u/{tw_id}`. See `CLAUDE.md → Smart Public Profile Router Rules §7`.

---

### 5. Session / localStorage Cache
**Purpose:** Client-side session cache. Stores user object after login.
**Source of Truth:** `localStorage.tw_user` (user object) · `localStorage.tw_jwt` (JWT token)
**Details:** `CLAUDE.md → Session Management` · `CLAUDE.md → Auth Gateway Rules §6`
**Do not recreate:** localStorage is a cache, not an authority. Do not gate security decisions on it. Do not add a second session key.

### 20c. Company Saved Candidates
**Purpose:** Private list of employee profiles saved by a company owner. Supports pipeline status tracking, notes, job linking, filtering, search, sorting, and statistics.
**Source of Truth:** `company_saved_candidates` (pipeline record) + `company_candidate_job_refs` (job link source) · `auth.py → save/remove/get/update/filter/stats functions` · `server.py → /company/saved-candidates/*`
**Responsible files:**
- `auth.py` — `_migrate_company_saved_candidates()`, `_migrate_company_candidate_job_refs()`, `save_company_candidate()`, `remove_company_candidate()`, `get_company_saved_candidates()`, `get_company_saved_candidates_count()`, `get_company_saved_candidates_filtered()`, `get_company_saved_candidates_stats()`, `update_company_saved_candidate()`, `VALID_CANDIDATE_STATUSES`, `CANDIDATE_STATUS_LABELS`, `VALID_CANDIDATE_SORTS`
- `server.py` — 6 endpoints + `_require_company_owner()` helper + `UpdateSavedCandidateInput` Pydantic model
**Details:** `ARCHITECTURE.md §55a` (base) · `ARCHITECTURE.md §55c` (Phase 6A — pipeline management) · `ARCHITECTURE.md §55e` (Phase 7A — filters & stats) · `ARCHITECTURE.md §55f` (Phase 7B — frontend filters UI)
**Auth:** JWT mandatory, `user_type='co'` only. `company_id` derived from `token["user_id"]` — no cross-company access.
**Two-table architecture:** `company_saved_candidates` = general pipeline record (one row per candidate per company). `company_candidate_job_refs(company_id, candidate_id, job_id)` PK = company-intentional job link (one row per candidate-job pair per company). The distinction: applying to a job ≠ being saved/classified by the company for that job. Only `company_candidate_job_refs` records the company's intentional association.
**Privacy:** Returns `candidate_id, tw_id, full_name, avatar_url, profession, city, country, job_id, status, notes, created_at, updated_at, job_titles[]` only. `job_titles[]` = job titles from `company_candidate_job_refs → jobs` (company-intentional links only — NOT all applied jobs). `job_id` in `company_saved_candidates` is legacy first-save origin (never overwritten, never shown in UI). Never returns email, phone, or KYC data.
**Pipeline statuses:** `saved | shortlisted | contacted | interview | hired | rejected`. Validated via `VALID_CANDIDATE_STATUSES`. `notes` max 500 chars. Per-job accepted state is tracked via `job_applications.status='accepted'` (returned as `per_job_accepted` field when `?job_id=` filter active).
**Filters (Phase 7A):** `status`, `job_id`, `unlinked`, `q` (search), `sort` (6 options via `VALID_CANDIDATE_SORTS`). `job_id` filter uses `EXISTS (SELECT 1 FROM company_candidate_job_refs ...)` — not `sc.job_id =`. `unlinked` filter uses `NOT EXISTS` on refs. `job_id` + `unlinked=true` are mutually exclusive (400). `q` max 80 chars. Stats `with_job`/`unlinked` counts use refs subqueries for consistency with filter results.
**Stats endpoint:** `GET /company/saved-candidates/stats` → `{ total, by_status:{...}, with_job, unlinked }`. All 6 statuses zero-filled. No candidate data returned.
**Badge:** `/count` endpoint always returns unfiltered total for badge display.
**Frontend (Phase 4):** `#candidatesBtn` in jobs toolbar (owner-only) + `#coCandidatesModal` + `.co-cand-*` CSS.
**Frontend (Phase 6B):** Pipeline manage panel inside saved tab. `.co-cand-saved-card` / `.co-cand-manage-panel` / `updateSavedCandidate()` / `_applyCardUpdate()`.
**Frontend (Phase 7B):** Filter bar inside `#coCandSavedShell`: 8 chips (الكل + 6 statuses + بدون وظيفة), search input (300ms debounce), sort select (6 options). Stats loaded via `getSavedCandidatesStats()`. Badge updated from `stats.total` (unfiltered). Load more button when `has_more=true`. Cards render in `#coCandSavedList` (not `_body`). Stats refresh after DELETE and PATCH. Filter-mismatch cards hide after PATCH. 8 empty state messages per filter.
**Frontend (UI Polish — PR fix/candidate-modal-ui-polish):** Custom dark inline pickers replace native `<select>` for status and job in manage panel and sort in filter bar. Chips: 4×2 grid, `44px` height, `9px` border-radius, per-status color always visible (not only on active). Manage button: neutral slate. Status badge colors unified with chip palette. See `ARCHITECTURE.md §55g` for full spec, picker DOM contract, value contracts, and forbidden patterns.
**Frontend (Option B — job_titles display):** Saved candidate cards show `.co-cand-job-chips` (up to 3, +N overflow) sourced from `item.job_titles[]`. Raw `job_id` is never displayed. In the applicants view, `.co-app-saved-ctx` shows "محفوظ · أيضاً في: [job names]" when `a.other_job_titles.length > 0`. Both CSS classes live in `static/company/company.css`.
**Do not recreate:** Do not add `company_id` to query/body. Do not search inside `notes`. Do not use raw user input in ORDER BY — use `VALID_CANDIDATE_SORTS` dict only. Do not change `/count` endpoint behavior. Do not update badge from `res.data.count` — use `stats.total` only. **Do not overwrite `job_id` in save or promote UPSERTs.** **Do not use `job_applications` to derive `job_titles[]`** — that includes all applications, not company-intentional links. **Do not display raw `job_id` in the UI** — use `job_titles[]` from `company_candidate_job_refs → jobs`. **Do not use `sc.job_id =` or `sc.job_id IS NULL` for job_id/unlinked filters** — use EXISTS subqueries on `company_candidate_job_refs`. **Do not create a second table for company-job linking** — `company_candidate_job_refs` is the only approved junction. **Do not derive per-job promoted state from `company_saved_candidates`** — use `job_applications.status='accepted'` and the `per_job_accepted` API field.
**Promote entry point (§66):** Candidates are promoted into this table via `POST /jobs/applications/{app_id}/promote` (not a direct `POST /company/saved-candidates`). This is the only atomic entry point that simultaneously marks the application `accepted`. See §15 (Job Applications) and `ARCHITECTURE.md §66`.
